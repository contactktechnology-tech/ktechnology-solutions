# Security Policy

## Supported Versions

Security fixes are provided for the most recent version of this repository and its actively maintained components.

If you are running older snapshots or forks, you are responsible for updating and applying fixes.

## Reporting a Vulnerability

If you believe you have found a security vulnerability, please report it responsibly.

1. Do not open a public issue describing the vulnerability.
2. Send a private report by email to:

contactktechnology@gmail.com

3. Include the following:
   - A clear description of the issue and potential impact
   - Steps to reproduce (proof of concept if safe to share)
   - Affected files, modules, or paths
   - Suggested remediation (if known)
   - Your contact details for follow-up

## Disclosure Process

After receiving a report, we will:
1. Acknowledge receipt as soon as practicable.
2. Assess severity and scope.
3. Prioritise a fix and validate remediation.
4. Coordinate disclosure timing where appropriate.

We may request additional details to reproduce or validate the issue.

## Safe Harbour for Good-Faith Research

We support good-faith security research intended to improve safety.

Permitted actions include:
- Testing against your own systems and accounts
- Minimal, non-destructive proof-of-concept demonstrations

Not permitted:
- Denial-of-service, service disruption, or resource exhaustion
- Accessing, modifying, deleting, or exfiltrating data that is not yours
- Social engineering, phishing, or physical attacks
- Scanning or testing third-party systems without explicit authorisation

## Security Expectations

You must not use this repository, its outputs, or derived works to:
- Create or enable malware, spyware, credential theft, or unauthorised access
- Bypass legal, contractual, or regulatory obligations
- Facilitate harassment, surveillance, or privacy-invasive behaviour

Any such use is prohibited.

## Credits

If you would like to be credited for a responsible disclosure, please indicate this in your report.
